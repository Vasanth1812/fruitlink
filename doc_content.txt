This version exists because the team's stronger skillset is Java/Spring, which is a legitimate and sufficient reason on its ownModular monolith first: one Spring Boot application, packages organized by domain (com.fruitlink.orders, com.fruitlink.ledger, ...), each module's repository layer accessed only through its own service — not by another module querying its tables directlyEvery write to money-adjacent data (credit limits, ledger entries, tax invoices, credit notes) goes through a @Transactional service method with validation and an audit-log write in the same transactionScheduled/background work is queue- or scheduler-driven, never blocking the request threadSpring Security + method-level @PreAuthorize is the authoritative RBAC boundary; the frontend hiding a control is UX only2. API DesignREST over JSON, versioned under /api/v1, resource-oriented URLs with @RestController + @RequestMappingDTOs annotated with Jakarta Bean Validation (@NotNull, @Positive, etc.); a @ControllerAdvice global exception handler maps domain exceptions (CreditLimitExceededException, StockUnavailableException) to consistent HTTP responses and error codesMonetary values as long integers in paise, never floating point, to avoid rounding drift in GST calculationsspringdoc-openapi publishes the live contract at /swagger-ui.html, kept in sync automatically with the controller annotations5. Authentication & AuthorizationSpring Security filter chain validates the JWT on every request; short-lived access token (~15 min) plus a rotated refresh token stored hashed in the databaseSalesman/Driver mobile refresh tokens bound to a device identifier to prevent replay from a different deviceRBAC enforced with method-level @PreAuthorize("hasPermission(...)") backed by a custom PermissionEvaluator that checks the caller's role against the module/action matrix, cached in Redis to avoid a DB roundtrip per request — this is the authoritative boundary, independent of anything the frontend hides6. Real-Time LayerSpring's WebSocket support with STOMP messaging, using a full-featured broker relay (RabbitMQ's STOMP plugin, or a Redis-backed alternative) rather than the simple in-memory broker, so multiple application instances can broadcast to the same connected clients.Fleet tracking: driver app publishes location updates to a topic; subscribed Super Admin map views receive them, with a delay flag raised if a vehicle drifts more than 20 minutes off its planned routeLive order board and notification tray follow the same publish/subscribe pattern, scoped by warehouse or user7. Scheduled & Background JobsJobTriggerNotesOrder window consolidationQuartz cron, 11:00 PM dailyAggregates shop orders into warehouse fulfillment instructionsRoute optimizationQuartz cron, 4:00 AM dailyComputes manifests; must complete before the 5:30 AM dispatch windowSafety-stock PO generationEvent-driven (RabbitMQ listener) on stock-threshold breachGenerates and routes a PO to the mapped vendorNotification dispatchEvent-drivenRoutes a triggered alert to push/SMS/WhatsApp per the notification matrixAging-receivables sweepQuartz cron, dailyRecalculates aging buckets and flags hard-block breachesQuartz is used over plain @Scheduled specifically for the two time-boxed jobs (order consolidation, route optimization) because it persists job state and handles misfires — if the app restarts at 3:59 AM, the 4:00 AM job still needs to fire reliably.8. Security, Validation & Audit LoggingTLS everywhere; secrets in AWS Secrets Manager, injected via Spring Cloud AWS or environment variables at deploy time — never committed to sourceEvery credit-limit change, price override, permission change, and claim approval writes an immutable audit-log row inside the same @Transactional service method that performs the changeRate limiting (e.g., Bucket4j) on auth and OTP-request endpointsJakarta Bean Validation on input DTOs; a separate response DTO layer (not the JPA entities directly) so internal-only fields never leak into API responses9. CI/CD PipelineGitHub Actions (or GitLab CI): mvn verify (unit + integration tests) → build Docker image → push to registry → deploy to staging → manual promote to productionFlyway migrations run as an explicit pipeline step against a target environment, never applied ad hocStaged environments: dev → staging → production, matching the frontend pipeline already documented12. Monitoring, Logging & AlertingMicrometer metrics exported to Prometheus/Grafana (or CloudWatch), covering JVM health alongside business metrics (order throughput, job duration)Structured JSON logs via Logback + logstash-logback-encoder, shipped to CloudWatch Logs or a hosted aggregatorSentry's Java SDK for exception tracking across the API and the Quartz/RabbitMQ job workersDedicated alerting on the two time-boxed jobs — a silent failure there has same-morning delivery consequences13. Open DecisionsMaven vs. Gradle — either works; pick whichever the team already standardizes on for other Java projectsRabbitMQ vs. Kafka for the async/event layer — RabbitMQ is simpler to operate at FruitLink's current volume; Kafka is worth it only if event throughput or replay requirements grow significantlySingle-tenant vs. multi-tenant — same open question as the Node version: if multi-tenant hosting becomes a real requirement, add an organization_id across the schema before real data accumulatesAWS managed stack vs. self-hosted VPS — same tradeoff as before: managed convenience and data residency vs. early-stage costRTO/RPO targets for backup/DR need a business-side answer
